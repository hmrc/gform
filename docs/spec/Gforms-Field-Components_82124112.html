This is the specification for all field components.  Examples of UI rendering are included, but for a UI-centric view please see the gforms UX catalog.

textText box

A text field.

The required format of the text field can be defined using format="formatName" to validate what is permitted to be entered in the text box. Refer to the text format reference for information about the types of format rendering and validation available. If the formatting rules are violated the UI should show the message given in errorMessage if provided, else it should revert to the default for the format used.

The text box can be pre-populated with a value using value="xxx". Expressions can be used here to set the value dynamically based on the user session context.

The text box can accept multi-line input via use of the multiline config option.

(O) value

(O) format (a valid text format type)

(O) round (up | down | ceiling | floor | halfUp | halfDown | halfEven* | unnecessary) - for fields with calculated values, see this table for rounding mode descriptions and examples. Note the default precision for calculations is 34 decimal digits. The precision of the value to be set can be defined using format=number(i,n) (see Text Formats).

(O) multiline (yes|no*)

 

Note: for all boolean values, use 'yes' and 'no' rather than 'true' and 'false', or accept both variants. True/false is not intuitive for some non-technical users.

Standard text box with no helpText:

"type": "text"
"id": "nameOfPremises"
"label": "Name of premises"

Text box with format="sterling" and helpText set:

"type": "text"

"label": "Standard rate IPT due for this 
period"
"format": "sterling"
"helpText": "You should deduct any standard 
credits that are due to you"

A text box whose value is set to a value from the user context and made read-only. Some help text is given to explain:

choice

Choice selection

Presents a number of choices that the user must select from.

The choices can be rendered by UIs in different ways depending on the options set.

Choice components can also be used when there is only one choice, for example where a declaration needs to be checked.

(O) value - an integer, or if multivalue is set, then a list of integers

(O) choices (list) - mandatory unless format=yesno

(O) format (vertical* | horizontal | yesno | inline)

(O) multivalue (yes|no*) - if yes then allows user to select multiple options. Note this also allows the user to specify zero values, but only if mandatory=no.

(O) optionHelpText (list) - Provides help text specific to each option. If provided, the length of the optionHelpText list must match the length of the choices list. Use an empty string for those options that do not have help text.

dateDate pickerAllows the user to enter a valid date.

(O) value - one of:

  • A full W3 date (YYYY-MM-DD)
  • today
  • next-MM-DD - sets value to the next occurrence of the given month and day-of-month - MM-DD must be a valid month and date.
  • last-MM-DD - sets value to the previous (or today's) occurrence of the given month and day-of-month - MM-DD must be a valid month and date.
  • an expression pointing to another date value

(O) offset (+|-n[d|m|y]) - number of days, months or years to add to the given value. E.g. +1d for tomorrow, -1d for yesterday, +3m for the same date in 3 months time.

(O) format - specify valid dates as per below; may be a comma separated list of formats:

  • anyDate* (default)
  • after | before | precisely <space>
    today | YYYY-MM-DD | next-MM-DD | previous-MM-DD | YYYY-MM-firstDay | YYYY-MM-lastDay | $otherFieldId
    <space>
    +|-n[d|m|y]

    Note: "YYYY", "MM" and "DD" match any year, month or date; alternatively any of these can be replaced by an actual year, month or date in order to provide a mask e.g. YYYY-04-DD for any day in April.

    Note: "DD" in a date format can also be replaced with "lastDay" to match the last day in the given month&year.

addressAddress entry

Allows the user to enter an address, which is captured as a fixed set of input fields (see value).

(O) value - JSON object containing values for the following keys:

  • (M) line1
  • (O) line2
  • (O) line3
  • (O) line4
  • (M*) postcode
  • (O*) country
    (*) note if international is yes then postcode must not be provided and country must be provided

(O) postcodeLookup (yes*|no) - if yes then the UI should offer to lookup the address from a postcode and address name/number provided by the user. Cannot be yes if international is yes.

(O) promptCopyFrom - if provided, (value is the ID of another address field in the form), then user should asked if they would like to use the address they've given in the given address field.

(O) promptCopyFromText - provide with promptCopyFrom to specify the text to display to the user when offering the option to copy an address from another address field.

(O) international (yes|no*) - if yes then country is required and postcode is not permitted. If no then country is not permitted and postcode is required. Cannot be yes if postcodeLookup is yes.

infoInformation messageProvides information to be shown to the user.

(O) label - if provided specifies a title for the information.

(M) infoText - the information message to show. Use markdown syntax for text formatting.

(O) infoType - the type of the information - one of:

  • standard (*default) - indicating info to be presented in the standard info style.
  • noformat - indicating info to be presented as standard page text with no special style.
  • long - UIs may consider showing this separate to the main form in some way, e.g. collapsing this text beneath the label or showing in pop-up.
  • important - indicating info to be presented as a special info style that draws greater attention to the message.
  • banner - indicating very important info to be presented in a banner.

groupField group

Groups together an enclosed set of fields, optionally with a label and help text.

Group components can be configured to capture repeating groups of field data. (Note whole sections can also be configured to repeat in a similar way.)

(M) fields (array of field objects) - a list of the fields to be grouped under the given id and label.

(O) format (horizontal|vertical*) - specifies for a UI whether to display the fields in the group horizontally or vertically. Note - this setting will be moved to presentationHint in a future release.

(O) repeatsMax (integer) - optionally, the maximum number of repeating groups permitted on the form. Indicates that this group is a repeating group. Can be set to an expression. Note that $n_ will implicitly be prefixed to the ID of every field in the repeating group.

(O) repeatsMin (integer) - if repeatsMax is set, repeatsMin can optionally also be set to specify the minimum number of repeats of the group required. Defaults to 1. Can be set using an expression. Cannot be set unless repeatsMax is also set. Note: setting this to 0 makes the repeating group optional. Setting this to the same value as repeatsMax requires the user to complete a fixed number of repeats.

(O) repeatLabel - if repeatsMax is set, the group's label is used as header text for the repeating group and repeatLabel is used as the header text for each repeating group. Use $n to represent the 0-based-index of each repetition. For example repeatLabel="Previous address $n" would be rendered as "Previous address 1" for the first repetition, "Previous address 2" for the second, and so on.

(O) repeatAddAnotherText - if repeatsMax is specified then this can be used to specify the text shown to the user when they are given the option to add another repetition of the group.

file

File upload

Allows user to upload a file for use as an attachment to the submission.

The file component is designed to be grouped with other components and as such has no label or helpText.

The upload mechanism and how the file is packaged with the submission is determined by the destinations defined in the form template. There must be at least one destination configures that supports file uploads (e.g. hmrcDMS).

 

(O) value - the value of a file component is a JSON object containing:

  • fileId - the identifier for the file returned by the file store (e.g. the fileId returned by FUaaS)
  • filename - the filename of the uploaded file
  • sha256hash - the SHA-256 hash of the contents of the uploaded file (note this is required for non-repudiation)

(O) metadata - a JSON object containing the metadata to send alongside the uploaded file. Expressions can be used within the metadata. Its content, and whether it is required, depends on the destination:

  • For hmrcDMS, this is a "metadata" object as defined in the FUaaS Put File Metadata API spec.

(O) allowContentTypes - an array of HTTP content type name strings. If provided, the uploaded file must have a content type that matches one of the types given in this array.

(O) maxSizeKb - if provided, the uploaded file must be less than the size given here (in Kb). If not provided, this reverts to the default limit imposed by the configured destination(s).

 

Standard example:

"type": "file",
"id": "reportFile",
"allowContentTypes": ["image/png",
 "application/pdf"],
"metadata": {
  "origin": {
    "utr": "$utr",
    "ggId": "$user.Id", 
    "submitterEmail": "$emailAddr" 
  },
  "sender": {
    "service": "EAOT"
  }
}

Prior to upload, the file component just shows a Browse button:

Once the user has clicked Browse and selected a file, the Browse button is replaced with the filename and a button/link to remove.

sensitive

Use to mark a field as sensitive. Data in sensitive fields should not be persisted other than as required to implement the submission journey. This data will therefore not be available for later retrieval or analysis:

  • sensitive fields cannot be used in a value expression that references any form instance other than itself.
  • Data in sensitive fields will not be sent to Google Analytics.
  • Data in sensitive fields will not be logged.

Most fields default to not sensitive (see right). The form author must ensure that any data that identifies an individual, such as name, address or identifiers like VAT reg number, or any card information, is marked sensitive.

OptionalIgnored for components that do not accept data i.e. info and group.

Text fields: default is false, except where:

  • format=ukBankAccountNumber
  • format=ukSortCode
  • format=utr
  • format=ukVrn
  • format=nino
  • format=telephoneNumber
  • format=email

Address fields: default is true

Choice fields: default is false

Info/group fields: default is false (n/a)

Date fields: default is false

File fields: default is false

"true"
validIf

Used to add validation to this field over and above any automatic validation (e.g. address fields have fixed maximum lengths,) and any validation imposed by component-specific options, notably format.

If set this can be any boolean expression, that is any expression that evaluates to true or false. If the expression evaluates to anything else, false is assumed. Often, an expression that checks the value of this same field is used, but validIf expressions can also check the value of any other field, and the boolean and conditional operators can be used to check multiple fields.

Optional-true

In this example declaration is an optional, multivalue choice component with only 1 possible choice (i.e. a single checkbox that can be ticked or unticked). This is common where the user must tick a box to agree, or untick to disagree.

"validIf": "$declaration == 1"

This would check that the "declaration" field, is set to value 1, i.e. it is ticked. One might then include errorMessage="You must acknowledge the declaration to proceed".

includeIf

If this option is provided then this field should only be included on a form if the given expression evaluates to true. See also the includeIf option for whole sections.

For API consumers, if this evaluates to false and yet a value for the field is given, the form should fail validation.

For UI consumers, if this evaluates to false the field should not be presented to the user.

This must be a predicate i.e. provide an expression that evaluates to true or false. If it does not evaluate to either true or false, the expression is invalid and the default (true) should be assumed.

Optional-

true

(i.e. the field is always included in a form of this type)

"${reclaimValue > 1000000}"
