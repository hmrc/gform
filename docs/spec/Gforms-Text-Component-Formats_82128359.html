Supported Field Formats

Notes on field formats:

  • While most formats can be boiled down to a text regex, bear in mind the format type also acts as a hint to the UI.  E.g. a UI may wish to insert a £ sign before any money field.
  • Regular expressions are used to limit the data entered, based on compatibility with back-end systems and security requirements.  Many of these derive from the DFS iForms Functions list  (also see DFS source code)

Basic Text

shortText *default

This is the default format if no format is specified.

Short text values much match the regular expression: [0-9a-zA-Z\s'\-]{0,1000}

This limits text values to letters a to z (upper and lower case), numbers, spaces, hyphens and apostrophes. A maximum of 1000 characters is allowed.

Testing 123 - valid

Testing*123 - invalid (contains invalid character)

TBC - e.g. Please use only letters a to z (upper and lower case), numbers, spaces, hyphens and apostrophes. A maximum of 1000 characters is allowed.
text

Text format is for fields that allow more comprehensive input and is therefore more permissive than shortText. Text must match the regular expression:

text<regex>

Constrains valid text to a regular expression (Java format) given inside the square brackets. Note this should be discouraged as most text field formats fall into one of the pre-defined formats in this table.

Important note: this applies the given regex after applying the standard text regex. I.e. it is impossible to loosen the validation further than text validation.

text<\w{8,10}> constrains field values to between 8 and 10 word characters (a-z, A-Z, 0-9, _).

Comments:

The email format regex is rather simplistic.  It is said to accept 99.99% of RFC-5322 addresses. That percentage though relies on the predominance of long TLDs, there is no 4 character limit in RFC-1034 .  The longest currently is 24, see:

curl -s http://data.iana.org/TLD/tlds-alpha-by-domain.txt | tail -n+2 | wc -L

It also does not allow RFC-6532 Internationalized email addresses.

Not all valid email address exist, so it would be better to have apermissive validator than a restrictive one.

There is no good reason why email validation needs to use a regex.  There is for example an existing public domain Scala library:

https://github.com/egulias/EmailValidator4Scala
